## Contribution Guidelines

### Code Contributions

1. **Fork the repository**
2. **Create a feature branch**: `git checkout -b feature/amazing-feature`
3. **Write tests** for your changes (90%+ coverage required)
4. **Follow code style** (ESLint + Prettier)
5. **Write clear commit messages**
6. **Update documentation** if needed
7. **Submit a pull request**

### Code Style

- Use ESLint and Prettier (configs provided)
- Write JSDoc comments for functions
- Keep functions small and focused
- Use meaningful variable names
- Follow existing patterns

### Testing

- Write unit tests for all new code
- Write integration tests for features
- Ensure all tests pass before submitting PR
- Aim for 90%+ code coverage

### Documentation

- Update README if adding features
- Add JSDoc comments to code
- Write tutorials for complex features
- Keep documentation up-to-date

## Types of Contributions

### Bug Reports

**Before submitting:**
- Check if the bug has already been reported
- Try to reproduce the bug consistently
- Gather relevant information (logs, screenshots, etc.)

**Include in your report:**
- Clear description of the bug
- Steps to reproduce
- Expected behavior
- Actual behavior
- Environment details (OS, browser, version)
- Relevant logs or error messages

### Feature Requests

**Before submitting:**
- Check if the feature has already been requested
- Consider if it aligns with project vision
- Think about implementation complexity

**Include in your request:**
- Clear description of the feature
- Use case and motivation
- Proposed implementation (if you have ideas)
- Potential challenges or concerns

### Documentation

Documentation is always welcome! Areas that need help:

- **Tutorials** - Step-by-step guides for common tasks
- **Examples** - Code examples and use cases
- **Translations** - Documentation in other languages
- **Clarifications** - Improving existing docs
- **Diagrams** - Visual explanations of concepts

### Research

We welcome academic research on:

- **Economic modeling** - Simulations of network behavior
- **Security analysis** - Cryptographic proofs and audits
- **Game theory** - Incentive mechanism analysis
- **Network theory** - Path finding optimization
- **Social science** - Trust and reputation dynamics

**Publishing research:**
- Share drafts in GitHub Discussions
- Submit papers to peer-reviewed journals
- Present at conferences
- Publish on arXiv or similar platforms

## Pull Request Process

### Before Submitting

1. **Update your fork** with latest main branch
2. **Run all tests** and ensure they pass
3. **Run linter** and fix any issues
4. **Update documentation** if needed
5. **Write clear PR description**

### PR Description Should Include

- **What** - What does this PR do?
- **Why** - Why is this change needed?
- **How** - How does it work?
- **Testing** - How was it tested?
- **Screenshots** - If UI changes

### Review Process

1. **Automated checks** run (tests, linting, coverage)
2. **Code review** by maintainers
3. **Feedback** and requested changes
4. **Approval** and merge

**Be patient** - Reviews may take a few days. We're all volunteers.
